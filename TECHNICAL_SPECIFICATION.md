# Техническое задание (ТЗ) на разработку MVP Telegram-бота «Traffk»

## 1. Концепция
**Название:** Traffk (окончательный вариант уточняется после проверки доступности username в Telegram).
**Тип продукта:** Telegram-бот (посредник-гарант в сделках между рекламодателями и арбитражниками).
**Цель MVP:** Запустить минимально работоспособный продукт для проверки гипотезы о востребованности сервиса на рынке. Фокус на безопасных сделках с фиксированной оплатой.
**Ключевая ценность:** Автоматизация и безопасность расчетов по принципу эскроу.

## 2. Пользовательские роли
1.  **Рекламодатель (Advertiser):** Создает задания, пополняет баланс, подтверждает выполнение.
2.  **Трафик (Traffic):** Просматривает ленту заданий, берет их в работу, получает уникальные ссылки, отмечает выполнение, выводит деньги.
3.  **Администратор (Admin):** Техническая роль для ручного управления спорами, пополнениями и выводами на этапе MVP.

## 3. Детальное описание функционала

### 3.1. Общий для всех пользователей функционал
*   **Регистрация/Авторизация:** При первом запуске бота пользователь регистрируется. Данные (`user_id`, `username`, `first_name`, `role`) сохраняются в БД.
*   **Главное меню:** Управление через инлайн-кнопки (`InlineKeyboardMarkup`).
*   **Баланс и транзакции:**
    *   Команда `/balance` отображает текущий доступный и замороженный баланс.
    *   Команда `/history` показывает историю транзакций (тип, сумма, статус, дата).
*   **Система уведомлений:** Уведомления о критических событиях (новое задание, запрос на подтверждение, поступление средств).

### 3.2. Функционал для Рекламодателя (Advertiser)
*   **Создание задания (`/create_task`):** Пошаговый сценарий (Finite State Machine - FSM):
    1.  **Название задания:** Произвольный текст.
    2.  **Исходная ссылка:** Целевой URL рекламодателя (валидация на корректность URL).
    3.  **Описание:** Текстовое поле для ТЗ (гео, запреты, крео и т.д.).
    4.  **Тип оплаты:** На MVP доступен только вариант «Фиксированная оплата».
    5.  **Сумма вознаграждения:** Числовое поле.
    6.  **Тип задания:** Выбор между «Публичное» (видно всем) и «Приватное» (с указанием `@username` конкретного трафика).
    7.  **Подтверждение:** Превью всех данных и кнопка «Создать».
    *   *Важно:* После подтверждения бот проверяет баланс. При успехе сумма вознаграждения замораживается (`frozen_balance`), задание активируется.
*   **Мои задания (`/my_tasks`):**
    *   Список созданных заданий с статусами (`active`, `pending_payment`, `completed`).
    *   Детальный просмотр задания: показывает список трафиков, взявших его в работу, и статус каждого.
    *   Для каждого трафика в статусе `pending_payment` отображается кнопка **«Подтвердить выплату»**.
*   **Пополнение баланса:** Кнопка «Пополнить баланс» выводит инструкцию и реквизиты для перевода (на MVP – номер карты Тинькофф/USDT-адрес). После перевода пользователь отправляет скриншот боту, админ вручную зачисляет средства.

### 3.3. Функционал для Трафика (Traffic)
*   **Лента заданий (`/feed`):**
    *   Список только публичных заданий со статусом `active`.
    *   Краткий вид: Название, Сумма, Гео (из описания).
    *   При клике на задание – полное описание и кнопка **«Взять в работу»**.
*   **Взятие задания:**
    *   При нажатии «Взять в работу»:
        1.  Бот проверяет, не брал ли пользователь это задание ранее.
        2.  Генерирует уникальную ссылку. **Техническое решение для MVP:** Использовать сервис Bitly или аналоги через API, либо простой редирект на своем сервере с параметром `?ref=<assignment_id>`.
        3.  Ссылка выдается пользователю.
        4.  Задание добавляется в раздел «Мои задания».
*   **Мои задания (`/my_assignments`):**
    *   Список заданий, взятых пользователем, с статусами (`in_progress`, `pending_payment`, `completed`).
    *   Для заданий в статусе `in_progress` есть кнопка **«Отметить выполненным»**. По нажатию статус меняется на `pending_payment`, а рекламодатель получает уведомление.
*   **Вывод средств:** Кнопка «Вывести средства» запрашивает реквизиты (номер карты/USDT-кошелек). Запрос уходит админу на ручную обработку.

### 3.4. Функционал Эскроу (Ядро системы)
*   **Создание задания:** Сумма вознаграждения сразу резервируется (`frozen_balance`) на балансе рекламодателя.
*   **Рабочий процесс:**
    1.  Трафик нажимает «Отметить выполненным» -> статус `assignment` меняется на `pending_payment`.
    2.  Рекламодатель получает уведомление в боте.
    3.  Рекламодатель в разделе `my_tasks` видит задание и нажимает «Подтвердить выплату» для конкретного трафика.
    4.  **Автоматический расчет:**
        *   Сумма списывается из `frozen_balance` рекламодателя.
        *   Сумма зачисляется на `balance` трафика.
        *   Статус `assignment` меняется на `completed`.
*   **Арбитраж:** Скрытая команда `/admin_arbitrage <assignment_id> <decision>` для принудительного разрешения споров.

## 4. Технические требования

*   **Стек технологий:**
    *   **Язык:** Python 3.11+
    *   **Фреймворк для бота:** Aiogram 3.x (асинхронный)
    *   **База данных:** PostgreSQL (предпочтительно) / SQLite (для самого простого MVP)
    *   **Хостинг:** VPS (Ubuntu 22.04 LTS)
    *   **Дополнительно:** Redis (для хранения состояний FSM и кэширования, сильно повышает отказоустойчивость и производительность).
*   **Безопасность:**
    *   Валидация всех входящих данных.
    *   Защита от SQL-инъекций (использование ORM или параметризованных запросов). Рекомендуется ORM `SQLAlchemy` или `asyncpg`.
    *   Проверка прав доступа к каждому обработчику сообщений и callback-запросов.
*   **Масштабируемость:** Код должен быть написан модульно, с учетом возможности добавления новых типов оплаты и интеграций.

## 5. Структура Базы Данных (Основные таблицы)

```sql
-- Таблица пользователей
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    tg_user_id BIGINT UNIQUE NOT NULL, -- ID пользователя в Telegram
    username VARCHAR(32), -- @username
    first_name VARCHAR(255),
    role VARCHAR(10) CHECK (role IN ('advertiser', 'traffic', 'admin')) NOT NULL,
    balance DECIMAL(15, 2) DEFAULT 0.00,
    frozen_balance DECIMAL(15, 2) DEFAULT 0.00, -- Замороженные средства
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Таблица заданий
CREATE TABLE tasks (
    id SERIAL PRIMARY KEY,
    advertiser_id INTEGER NOT NULL REFERENCES users(id),
    title VARCHAR(255) NOT NULL,
    original_url TEXT NOT NULL, -- Исходная ссылка рекламодателя
    description TEXT,
    reward DECIMAL(15, 2) NOT NULL, -- Вознаграждение
    type VARCHAR(10) CHECK (type IN ('public', 'private')) NOT NULL,
    status VARCHAR(20) CHECK (status IN ('active', 'paused', 'completed')) DEFAULT 'active',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Таблица связей "Задание - Трафик"
CREATE TABLE assignments (
    id SERIAL PRIMARY KEY,
    task_id INTEGER NOT NULL REFERENCES tasks(id),
    traffic_id INTEGER NOT NULL REFERENCES users(id),
    status VARCHAR(20) CHECK (status IN ('in_progress', 'pending_payment', 'completed')) DEFAULT 'in_progress',
    unique_link TEXT, -- Сгенерированная уникальная ссылка
    proof_text TEXT, -- Текст-подтверждение от трафика
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(task_id, traffic_id) -- Чтобы нельзя было взять одно задание дважды
);

-- Таблица транзакций (для истории)
CREATE TABLE transactions (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id),
    type VARCHAR(20) NOT NULL, -- 'deposit', 'withdrawal', 'freeze', 'payout', 'release' (разморозка)
    amount DECIMAL(15, 2) NOT NULL,
    status VARCHAR(20) CHECK (status IN ('pending', 'completed', 'cancelled')) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

## 6. План разработки и критерии приемки (Roadmap)

| Этап | Срок | Задача | Критерий приемки |
| :--- | :--- | :--- | :--- |
| **Этап 0** | 2 дня | Настройка окружения, сервера, БД, базовый каркас бота. | Бот запускается, отвечает на команду /start, регистрирует пользователя в БД. |
| **Этап 1** | 5 дней | Система балансов и транзакций. Интерфейс админа для ручного пополнения/вывода. | Админ может зачислить/списать средства пользователю через команду. |
| **Этап 2** | 7 дней | Полный FSM для создания заданий (с резервированием средств). Лента заданий (/feed). | Рекламодатель создает задание, деньги замораживаются. Трафик видит его в ленте. |
| **Этап 3** | 5 дней | Функционал взятия заданий, генерация ссылок, раздел "Мои задания". | Трафик берет задание, получает уникальную ссылку, видит его в "Мои задания". |
| **Этап 4** | 4 дня | Механика эскроу: "Отметить выполненным" -> "Подтвердить выплату" -> автоматический расчет. | Проходит полный цикл сделки от создания до выплаты без ошибок. |
| **Этап 5** | 2 дня | Тестирование, отладка, написание инструкции. | MVP готово к передаче 2-3 тестовым пользователям. |
